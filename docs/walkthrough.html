Act I · the model

Exactly the whiteboard — as seventeen migrations

Rich's sketch became the schema, unchanged in spirit: a paying client owns everything; projects (optionally grouped into portfolios) connect to contacts only through typed, dated relationships; users sign in, contacts don't; documents attach to projects and carry role gates. Row-level security wraps every table.

Category rule — nothing construction-specific in the schema. Types and lists live as data rows; a category-specific attribute proposal stops the line for a design conversation first (the Rubicon rule). Nothing named "tenant" exists: a tenant is a contact in a time-bound lessee relationship — turnover is a date, history is retained, reactivation is one edit.

See everything, act inline, act in bulk

  • Projects · Contacts · Groups · Users — dense grids, no drill-down mazes
  • Reconciliation chip on every grid: "8 of 8 expected" — a gap is an amber flag, never a silent miss
  • Bulk vendor reassign across projects in one transaction — the exact scenario from the call
  • Detail rail: raw + normalized address, lease windows, gated Journey and Budget slots
03
frictionless intake · merged

Four ways in, one review pattern

  • Spreadsheet import — full tree in one drop: groups, addresses, projects, contacts, leases
  • iPhone contact card (VCF), address type-ahead, and natural-language quick capture
  • Always preview → commit; never auto-commit; re-importing the same file is idempotent
  • Leverage stat after every import: records created per minute of operator effort
04
financial truth + journey · merged

The proven primitives, ported onto the core

  • Double-entry ledger — the database itself rejects an unbalanced write
  • Budget rail: post an expense, approve a change order, undo by reversal — never deletion
  • Journey: a scrubbable project timeline driven by events and relationship windows
  • Documents upload with role gates enforced on list and download
Act III · the receipts

Proof, not promise

14/14
sql tests · hosted

The full suite — isolation, lease time-bounds, reconciliation, module gating, ledger invariants — green against the dedicated dev database, not just a laptop.

Σ=0
ledger invariant

Debits equal credits is a deferrable database constraint. An unbalanced write throws; corrections are reversals, so the audit trail is never rewritten.

401
fails closed

The public URL shows an unauthenticated visitor nothing — every API refuses before touching data. Sign-in (Auth0) is the next founder unlock, and the door stays shut until then.

×0
prod isolation

The engine cannot touch the shared production database: the connection layer hard-refuses its project ref at boot, and the deploy script refuses to ship env that resolves to it.

1:1
provenance

Every mutation writes an event — actor, source, timestamp, duration. Cost-to-serve and dispute protection come from the same append-only stream.

＝
idempotent intake

Dropping the same spreadsheet twice updates instead of duplicating. Unknown contact types land as drafts for review — never a silent new column.

Two grants, one project — enforced in the database

The same project seen by an admin and by a read-only grant with budgets switched off. The budget rail, the money moments, and the gated agreement vanish — and not because the UI hides them: the rows never leave Postgres. The lens pass caught a real leak (money surfacing in the journey for budget-less grants) before merge; that story is worth telling in the room.

The four claims, made testable

Frictionless

  • Zero required data entry on day one — we import, the client verifies
  • Spreadsheet to populated workspace: instrumented against a <10-minute bar
  • New project + contacts by typed sentence or contact card in under a minute

Scalable

  • Category-agnostic core — property management today, any vertical as data rows
  • Serverless app + managed Postgres; the schema is the product, surfaces are cheap
  • Cost-to-serve measured per action from day one via the events stream

User-friendly

  • Grid-first: recurring actions never hide behind navigation
  • Bulk operations are day-one features, not "advanced"
  • Reconciliation chips make trust visible on every screen

Risk-free

  • Row-level security on every table, proven by tests, not asserted
  • Fail-closed public surface; append-only provenance; reversal-not-delete money
  • Old app frozen; one write lane; every merge founder-reviewed and browser-dogfooded
